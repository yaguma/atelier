# 計画ルール

## 基本原則

計画は軽量に、実装は素早く。過度な計画はコンテキストを消費し、計画自体が陳腐化するリスクがある。

---

## 計画の粒度

### 5箇条書き以内ルール

計画は**最大5箇条書き**に収める。各項目は1行で表現できる粒度にする。

```
# OK: 簡潔な計画
1. quest-generator.tsにフィルタ関数を追加
2. QuestList.tsでフィルタUIを実装
3. ユニットテストを追加
4. 統合テストで動作確認

# NG: 過度に詳細な計画
1. quest-generator.tsを開き、現在の関数一覧を確認する
2. filterQuestsByDifficulty関数のシグネチャを設計する
3. 引数にQuestDifficulty[]を受け取るようにする
4. 戻り値はQuest[]とする
5. テストを先に書く
6. RED状態を確認する
7. 実装を書く
8. GREEN状態を確認する
9. リファクタリングする
10. QuestList.tsのUIにフィルタボタンを追加する
... (以下省略)
```

---

## 拡張計画モードの禁止

以下のパターンは禁止する。

- 計画段階で長時間のリサーチを行う
- 計画を完璧にしてから実装に着手しようとする
- 不明点を全て解決してから実装を始める

### 代わりにすべきこと

- 不明点は**実装しながら解決**する
- 計画より**プロトタイプ実装**を優先する
- 大きなタスクは**小さなステップに分割**して順次実行する

---

## 計画が必要なケース

全てのタスクに計画が必要なわけではない。以下の場合のみ計画を立てる。

| ケース | 計画要否 |
|--------|----------|
| 単一ファイルの修正 | 不要（直接実装） |
| 3ファイル以下の変更 | 不要（直接実装） |
| 複数機能にまたがる変更 | 簡潔な計画（5箇条書き以内） |
| アーキテクチャ変更 | 簡潔な計画 + ユーザー確認 |

---

## 禁止事項

- 5箇条書きを超える計画の作成
- 計画段階での長時間リサーチ
- 計画の完璧化を理由にした実装の先延ばし
- 実装着手前の全不明点の解消
