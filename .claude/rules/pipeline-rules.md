# パイプラインルール

## 概要

PR作成からマージまでの標準パイプラインフローを定義する。品質ゲートを設けることで、一貫した品質を担保する。

---

## 標準パイプラインフロー

```
実装完了 → CIチェック → PR作成 → レビュー → 指摘修正 → 再レビュー → マージ
```

### Step 1: 実装完了確認

PR作成前に以下を全てパスさせること。

```bash
pnpm test -- --run    # ユニットテスト
pnpm typecheck        # 型チェック
pnpm lint             # リントチェック
```

全てパスしない場合、PR作成に進まない。

### Step 2: PR作成

- Conventional Commits形式のタイトル
- Summary（変更内容の箇条書き）とTest Plan（テスト項目）を記載
- 関連するIssue番号をリンク

### Step 3: レビュー

- PR作成後、レビューを**必ず実施**する
- レビューをスキップしてマージすることは禁止
- レビュー前にコンテキストが肥大化している場合は、compactを実行してから臨む

### Step 4: 指摘修正

レビュー指摘は重要度に応じて対応する。

| 重要度 | 対応 |
|--------|------|
| **Critical** | マージ前に必ず修正 |
| **Warning** | 修正を強く推奨（修正またはIssue作成で対応） |
| **Info** | 任意対応（修正、Issue作成、またはスキップ） |

### Step 5: 再レビュー（必要に応じて）

- Critical指摘の修正後は再レビューを実施
- Warning/Info指摘のみの修正は、修正内容の確認のみでよい

### Step 6: マージ

- 全てのCIチェックがパスしていること
- Critical指摘が全て解決していること
- mainブランチへのマージはPR経由で行う

---

## レビュー前の準備

### コンテキストcompact

長時間の実装作業後はコンテキストが肥大化している。レビュー精度を維持するため、以下を推奨する。

- 実装作業が長時間に及んだ場合、レビュー前にコンテキストをcompactする
- compactにより、レビューエージェントが必要な情報に集中できる

### セルフチェック

レビュー依頼前に、以下のセルフチェックを実施する。

- [ ] 全テストがパスしているか
- [ ] 型エラーがないか
- [ ] リントエラーがないか
- [ ] 不要な`console.log`が残っていないか
- [ ] コミットメッセージがConventional Commits形式か

---

## 禁止事項

- レビューをスキップしてPRをマージする
- Critical指摘を未修正のままマージする
- CIチェックが失敗した状態でマージする
- レビュー指摘を無視して対応理由を記載しない

---

## レビューループの上限

レビュー→修正→再レビューのループは**最大5回**とする。5回に達した場合は手動での判断を仰ぐ。
