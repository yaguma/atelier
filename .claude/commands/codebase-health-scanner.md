description: コードベースの健全性指標をスキャンし、レポート生成・Issue自動作成を行う
allowed-tools: Read, Glob, Grep, Bash, Write, Edit

コードベースの健全性指標（未使用export、循環依存、any型使用、デッドコード、マジックナンバー、大きすぎるファイル）をスキャンし、レポートを生成する。閾値を超過した場合はGitHub Issueを自動作成する。

# context

ソースディレクトリ="atelier-guild-rank/src"
テストディレクトリ="atelier-guild-rank/tests"
レポート出力先="docs/reports"
日付={{YYYY-MM-DD形式の今日の日付}}
レポートファイル="docs/reports/health-scan-{{日付}}.md"
オプション=$1
スキャン結果={}
ヘルススコア=100
前回レポート=null

# 閾値定義

| 指標 | 黄色（Warning） | 赤色（Critical） |
|------|----------------|-----------------|
| any型使用 | 5箇所以上 | 10箇所以上 |
| 循環依存 | 1件以上 | 3件以上 |
| 未使用export | 10件以上 | 20件以上 |
| 300行超ファイル | 5ファイル以上 | 10ファイル以上 |
| マジックナンバー | 10箇所以上 | 20箇所以上 |
| デッドコード（未参照ファイル） | 3ファイル以上 | 5ファイル以上 |

# ヘルススコア計算

100点満点から以下の減点を行う:
- any型使用: 1箇所あたり -2点
- 循環依存: 1件あたり -5点
- 未使用export: 1件あたり -1点
- 300行超ファイル: 1ファイルあたり -2点
- マジックナンバー: 1箇所あたり -1点
- デッドコード: 1ファイルあたり -3点
- 最低スコアは0点

# step

- オプションを解析する
  - `--skip-issue`: Issue自動作成をスキップ
  - `--report-only`: レポート生成のみ（Issue作成もトレンド比較もスキップ）
  - `--trend`: 前回レポートとのトレンド比較のみ表示
- `--trend` の場合は step7 へジャンプ
- step1 を実行する

## step1: スキャン準備

- ソースディレクトリの存在を確認する
- レポート出力先ディレクトリを作成する（存在しない場合）

```bash
mkdir -p docs/reports
```

- 前回のレポートファイルを検索して context の前回レポートに設定する

```bash
ls -t docs/reports/health-scan-*.md 2>/dev/null | head -1
```

- step2 を実行する

## step2: Phase 1 - スキャンエンジン実行

以下の6つのスキャンを順番に実行し、結果を context のスキャン結果に格納する。

### 2-1: any型使用検出

Grep ツールを使用して `src/` 配下の `.ts` ファイルから any 型の使用箇所を検出する。

検出パターン:
- `: any` - 型注釈での any 使用
- `as any` - 型アサーションでの any 使用
- `<any>` - ジェネリクスでの any 使用

```bash
# 検出コマンド例
grep -rn ": any\b" atelier-guild-rank/src/ --include="*.ts" --include="*.tsx"
grep -rn "as any" atelier-guild-rank/src/ --include="*.ts" --include="*.tsx"
grep -rn "<any>" atelier-guild-rank/src/ --include="*.ts" --include="*.tsx"
```

除外対象:
- `// biome-ignore` コメント付きの行
- `.d.ts` ファイル（型定義ファイル）
- テストファイル（`*.test.ts`）
- `node_modules/`

結果を `スキャン結果.any型` に格納する（ファイルパス、行番号、該当コードのリスト）。

### 2-2: 循環依存検出

インポートグラフを分析して循環依存を検出する。

手順:
1. Grep ツールで全 `.ts` ファイルの `import` 文を抽出する
2. 各ファイルのインポート先を解析し、依存グラフを構築する
3. DFS（深さ優先探索）で循環を検出する

```bash
# importの抽出例
grep -rn "^import.*from" atelier-guild-rank/src/ --include="*.ts" --include="*.tsx"
```

特に注意する循環パターン:
- `features/A` -> `features/B` -> `features/A`（機能間の循環）
- `shared/` -> `features/` -> `shared/`（レイヤー逆転）

結果を `スキャン結果.循環依存` に格納する（関与するファイルのリスト、循環パスの説明）。

### 2-3: 未使用export検出

exportされているがどこからもimportされていないシンボルを検出する。

手順:
1. Grep ツールで全 `export` 文を抽出する
2. 各exportされたシンボル名で Grep 検索し、import側での使用を確認する
3. import側で使用されていないexportを未使用と判定する

```bash
# export抽出例
grep -rn "^export " atelier-guild-rank/src/ --include="*.ts" --include="*.tsx"
grep -rn "export {" atelier-guild-rank/src/ --include="*.ts" --include="*.tsx"
grep -rn "export default" atelier-guild-rank/src/ --include="*.ts" --include="*.tsx"
```

除外対象:
- `index.ts`のre-export（公開API定義のため）
- `main.ts`（エントリーポイント）
- 型定義のexport（`export type`、`export interface`）で、同パッケージ内の型定義ファイルから参照されているもの

結果を `スキャン結果.未使用export` に格納する（ファイルパス、行番号、シンボル名のリスト）。

### 2-4: デッドコード検出

どこからも参照されていないファイル・関数を検出する。

手順:
1. Glob ツールで全 `.ts` ファイルを列挙する
2. 各ファイルが他のファイルからimportされているかを Grep で確認する
3. どこからもimportされていないファイルを未参照と判定する

```bash
# ファイル参照チェック例
# ファイル名からimportを検索
grep -rn "from.*['\"].*target-file['\"]" atelier-guild-rank/src/ --include="*.ts"
```

除外対象:
- `main.ts`（エントリーポイント）
- `index.ts`（re-exportファイル）
- シーンファイル（`*Scene.ts` - Phaserが動的にロード）
- 設定ファイル（`vite.config.ts`等）
- 型定義ファイル（`.d.ts`）

結果を `スキャン結果.デッドコード` に格納する（未参照ファイルのリスト）。

### 2-5: マジックナンバー検出

定数化されていない数値リテラルを検出する。

検出パターン:
- 関数の引数やオブジェクトリテラル内の数値（0, 1, -1, 2 等の一般的な値を除く）
- 条件式内の数値リテラル
- タイマーやアニメーション時間の直値

```bash
# マジックナンバー検出例（一般的な0, 1, -1, 2は除外）
grep -rn "[^a-zA-Z_][3-9][0-9]\+[^a-zA-Z_0-9x]" atelier-guild-rank/src/ --include="*.ts"
grep -rn "0x[0-9a-fA-F]\+" atelier-guild-rank/src/ --include="*.ts"
```

除外対象:
- `game-config.ts`（定数定義ファイル）
- `theme.ts`（テーマ定義ファイル）
- `constants/` ディレクトリ内のファイル
- 配列インデックス（`[0]`, `[1]`等）
- `enum` 定義内の数値
- テストファイル内のテストデータ

結果を `スキャン結果.マジックナンバー` に格納する（ファイルパス、行番号、該当コードのリスト）。

### 2-6: 大きすぎるファイル検出

300行を超えるTypeScriptファイルを検出する。

```bash
# 行数カウント
find atelier-guild-rank/src/ -name "*.ts" -o -name "*.tsx" | xargs wc -l | sort -rn | head -20
```

除外対象:
- 型定義ファイル（`.d.ts`）
- 自動生成ファイル
- テストファイル

結果を `スキャン結果.大きすぎるファイル` に格納する（ファイルパス、行数のリスト）。

- step3 を実行する

## step3: ヘルススコア計算

context のスキャン結果から以下の計算でヘルススコアを算出する:

```
スコア = 100
スコア -= any型使用数 * 2
スコア -= 循環依存数 * 5
スコア -= 未使用export数 * 1
スコア -= 300行超ファイル数 * 2
スコア -= マジックナンバー数 * 1
スコア -= デッドコードファイル数 * 3
スコア = max(0, スコア)
```

各指標の閾値判定を行い、状態（正常/Warning/Critical）を設定する。

- step4 を実行する

## step4: Phase 2 - レポート生成

Write ツールでレポートファイルを `docs/reports/health-scan-{{日付}}.md` に生成する。

- `--report-only` の場合はここで終了、結果をユーザに報告する
- step5 を実行する

## step5: Phase 2 - トレンド比較

前回レポートが存在する場合:
1. 前回レポートファイルを Read ツールで読み込む
2. 各指標の件数を抽出する
3. 増減を計算し、レポートのトレンドセクションとサマリーの前回比列を更新する

前回レポートが存在しない場合:
- トレンドセクションに「初回スキャンのためトレンドデータなし」と記載

- step6 を実行する

## step6: Phase 2 - Issue自動作成

`--skip-issue` が指定されている場合はスキップして step8 へ。

Critical判定の指標がある場合:
1. `gh issue create` で Issue を作成する
2. Issue のタイトル: `health: コードベースヘルスチェック警告 ({{日付}})`
3. Issue のラベル: `health`, `automated`
4. Issue の本文に Critical/Warning の指標と具体的な改善箇所を記載

```bash
gh issue create \
  --title "health: コードベースヘルスチェック警告 ({{日付}})" \
  --label "health,automated" \
  --body "$(cat <<'EOF'
## コードベースヘルスチェック結果

**ヘルススコア**: {{スコア}}/100

### Critical/Warning指標

{{Critical/Warningの指標一覧}}

### 対応が必要な箇所

{{具体的なファイル・行の情報}}

### レポート

詳細は `docs/reports/health-scan-{{日付}}.md` を参照。

---
このIssueはコードベースヘルススキャナーにより自動作成されました。
EOF
)"
```

Warning のみでCriticalがない場合:
- Issue は作成せず、レポートのみ生成する
- ユーザに警告レベルの問題を報告する

すべて正常の場合:
- Issue は作成しない
- ユーザに健全であることを報告する

- step8 を実行する

## step7: トレンド表示のみ

`--trend` オプション指定時の処理:

1. `docs/reports/` 配下の全 `health-scan-*.md` ファイルを時系列で取得する
2. 各レポートからヘルススコアと指標件数を抽出する
3. 時系列でのトレンドを表形式で表示する

```
日付       | スコア | any型 | 循環依存 | 未使用export | 300行超 | マジックNo | デッドコード
2026-03-01 | 85     | 3     | 0        | 8            | 4       | 7          | 1
2026-03-08 | 78     | 5     | 1        | 12           | 5       | 9          | 2
2026-03-15 | 82     | 4     | 0        | 10           | 5       | 8          | 1
```

- 処理完了後、ユーザに結果を報告する

## step8: 結果報告

ユーザに以下を報告する:
- ヘルススコアとその評価
- Critical/Warning の指標がある場合はその概要
- レポートファイルのパス
- Issue を作成した場合はその URL
- 改善の優先度が高い項目のトップ3

# CI統合指針

## /loop コマンドとの連携

定期的にヘルススキャンを実行する場合:

```bash
# 週次実行の例
claude code --skill codebase-health-scanner
```

## PRごとのヘルスチェック

PRのCI/CDパイプラインに組み込む場合:
1. PR作成時にヘルススキャンを実行
2. 前回（mainブランチ）のスキャン結果と比較
3. スコアが悪化した場合はPRコメントで警告

```bash
# PRヘルスチェック（Issue作成をスキップ）
claude code --skill codebase-health-scanner --skip-issue
```

## 推奨実行頻度

| 実行タイミング | オプション | 目的 |
|---------------|-----------|------|
| 週次（月曜日） | （オプションなし） | 定期的な健全性チェック + Issue作成 |
| PR作成時 | `--skip-issue` | PR差分による悪化検出 |
| スプリント振り返り | `--trend` | トレンド確認と改善計画 |
| リリース前 | （オプションなし） | リリース品質ゲート |
