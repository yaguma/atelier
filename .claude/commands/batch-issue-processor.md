# batch-issue-processor

## 目的

GitHub Issueの複雑度・依存関係を自動分類し、独立したIssueを並列処理するバッチプロセッサー。
3つのフェーズ（分析 → 並列処理 → 統合検証）で構成される。

## 前提条件

- `gh` CLI がインストールされ認証済みであること
- GitHubリポジトリ内で実行すること
- mainブランチが最新であること

## 事前準備

1. **追加ルールの読み込み**
   - `.claude/rules` ディレクトリ内のすべてのファイルを読み込み、追加ルールとして適用
   - `CLAUDE.md` を読み込み、プロジェクト固有のルールを適用

2. **現在のリポジトリ状態の確認**
   - `git branch --show-current` で現在のブランチを確認
   - `git status` で未コミットの変更がないことを確認
   - 未コミットの変更がある場合は警告し、続行するかユーザに確認

## 実行内容

### Phase 1: Issue分析エンジン

#### Step 1.1: Issue一覧の取得

- `gh issue list --state open --json number,title,body,labels,milestone,assignees --limit 100` でオープンIssueを取得
- $1 が指定されている場合、フィルタ条件を適用する
  - ラベル名の場合: `--label "$1"` を追加
  - マイルストーンの場合: `--milestone "$1"` を追加
  - 自然言語の場合: 取得後にタイトル・本文からキーワードマッチでフィルタ
- 取得したIssue数をユーザに報告する

#### Step 1.2: 複雑度の自動判定

各Issueの本文・タイトルから複雑度を自動判定する。

**判定基準:**

| 複雑度 | 基準 | 目安時間 |
|--------|------|----------|
| **S** | 設定変更、ドキュメント更新、1ファイル変更、タイポ修正 | 1-2時間 |
| **M** | 単一機能の追加・修正、テスト追加、小規模リファクタリング | 半日 |
| **L** | 複数ファイルにまたがる機能、新しいfeatureモジュール追加 | 1日 |
| **XL** | アーキテクチャ変更、大規模リファクタリング、横断的関心事 | 数日 |

**判定ロジック:**

1. **キーワードベース判定**（第一フィルタ）
   - S指標: `typo`, `docs`, `README`, `設定`, `config`, `.md`, `コメント`, `rename`
   - M指標: `追加`, `修正`, `fix`, `update`, `テスト`, `test`, `単一`, `コンポーネント`
   - L指標: `機能`, `feature`, `feat`, `複数`, `連携`, `統合`, `モジュール`
   - XL指標: `アーキテクチャ`, `リファクタリング`, `移行`, `migration`, `全体`, `基盤`

2. **ラベルベース判定**（第二フィルタ、ラベルが付いている場合は優先）
   - `size/S`, `size/M`, `size/L`, `size/XL` ラベルがあればそれを採用
   - `bug` ラベル → 基本M（本文の規模感で調整）
   - `enhancement` ラベル → 基本M-L
   - `refactor` ラベル → 基本L-XL

3. **本文の構造分析**（第三フィルタ）
   - チェックリストの項目数: 1-3項目→S-M、4-7項目→M-L、8項目以上→L-XL
   - 言及されているファイル数: 1ファイル→S-M、2-5ファイル→M-L、6ファイル以上→L-XL
   - セクション数（##の数）: 1-2→S-M、3-5→M-L、6以上→L-XL

#### Step 1.3: 依存関係の自動検出

以下の情報源から依存関係を検出する。

1. **本文中の参照**
   - `#数字` パターン（他のIssueへの参照）を検出
   - `depends on`, `blocked by`, `after`, `requires`, `前提`, `依存` などのキーワードを検出
   - コードブロック内のファイルパスから同一ファイルを変更するIssueを検出

2. **ラベルベースのグルーピング**
   - 同一ラベルを持つIssue群を関連グループとして検出
   - 同一マイルストーンのIssueを関連グループとして検出

3. **暗黙的依存の推定**
   - 同じfeatureモジュール（`features/quest/`, `features/alchemy/` 等）に影響するIssueを検出
   - 同じ共通サービス（`shared/services/`等）に影響するIssueを検出
   - アーキテクチャ変更（XL）は他のすべてのIssueの暗黙的前提と判定

#### Step 1.4: グルーピングと優先順位付け

1. **依存関係グラフの構築**
   - 有向グラフとしてIssue間の依存を表現
   - 循環依存がある場合は警告

2. **独立Issueの特定**
   - 他のIssueに依存せず、他のIssueからも依存されていないIssueを「独立Issue」として分類
   - 依存先がすべて完了済み（closed）のIssueも「独立Issue」として分類

3. **並列実行グループの決定**
   - 独立Issue群を複雑度でソート（S→M→L→XL）
   - 同時実行数の上限を設定（デフォルト: 5）
   - XLサイズは並列実行せず、単独で処理

4. **分析結果レポートの出力**
   - 全Issueの分析結果テーブル
   - 依存関係グラフ（Mermaid記法）
   - 並列実行グループの提案
   - ユーザに実行するIssueグループの承認を求める

### Phase 2: 並列処理エンジン

#### Step 2.1: 実行計画の確認

- Phase 1で生成した並列実行グループをユーザに提示
- 実行するIssueグループの承認を得る
- 除外したいIssueがあれば指定可能

#### Step 2.2: 並列処理の実行

承認されたIssueグループに対して、以下を並列実行する。

**各Issueの処理フロー:**

1. **ブランチ作成**
   - mainブランチから作業ブランチを作成
   - ブランチ名: `<type>/#<issue-number>-<description>`（git-workflow.mdに準拠）

2. **実装の実行**
   - Issue本文から実装内容を理解
   - コードベースを分析して変更箇所を特定
   - 実装を実行
   - テストの追加・修正
   - lint/formatの実行

3. **進捗報告**
   - 各Issueの処理開始時にユーザに通知
   - 処理完了時に結果を通知（成功/失敗/スキップ）

**並列実行の制約:**

- 同時実行数: 最大5つ（設定可能）
- Sサイズ: 最大5つ同時
- Mサイズ: 最大3つ同時
- Lサイズ: 最大2つ同時
- XLサイズ: 1つのみ（単独実行）

#### Step 2.3: 進捗モニタリング

- 全Issueの処理状況をリアルタイムで追跡
- 失敗したIssueの原因を記録
- 処理完了後にサマリーレポートを生成

### Phase 3: 統合・検証

#### Step 3.1: コンフリクト検出

- 並列実装された各ブランチの変更ファイルを比較
- 同一ファイルを変更しているブランチ組を検出
- コンフリクトの深刻度を評価
  - **高**: 同一関数・メソッドを変更
  - **中**: 同一ファイルの異なる箇所を変更
  - **低**: 関連ファイルを変更（直接的なコンフリクトなし）

#### Step 3.2: マージ順序の決定

- 依存関係グラフに基づいてマージ順序を決定
- コンフリクトの深刻度を考慮
  - コンフリクトが低いものを先にマージ
  - コンフリクトが高いものは手動解決を推奨

**マージ順序の決定ルール:**

1. 依存先のないIssue（ルートノード）を最初にマージ
2. Sサイズ → Mサイズ → Lサイズ → XLサイズの順
3. コンフリクトが少ないものを優先
4. 同一優先度の場合はIssue番号の昇順

#### Step 3.3: 統合テストの実行

- 各ブランチのマージ後に以下を実行:
  1. `pnpm typecheck` - 型チェック
  2. `pnpm lint` - リントチェック
  3. `pnpm test -- --run` - ユニットテスト
  4. `pnpm build` - ビルド確認

- テスト失敗時の対応:
  - 失敗内容をレポートに記録
  - 自動修正可能な場合は修正を試みる
  - 手動対応が必要な場合はスキップしてレポート

#### Step 3.4: 最終レポートの生成

処理結果の最終レポートを生成する。

**レポート内容:**

- 処理したIssue一覧と結果（成功/失敗/スキップ）
- 作成されたブランチ一覧
- コンフリクト検出結果
- 推奨マージ順序
- テスト結果サマリー
- 手動対応が必要な項目一覧

## rules

### Issue処理の原則

- **安全性優先**: 不明な場合はスキップして手動対応を推奨
- **原子的変更**: 各Issueの変更は独立してrevertできるようにする
- **テスト必須**: 実装変更には必ずテストを伴う
- **既存コードの尊重**: プロジェクトのコーディング規約・アーキテクチャに従う

### 並列処理の制約

- 同一ファイルを変更する可能性が高いIssueは並列実行しない
- XLサイズのIssueは必ず単独で処理する
- 失敗したIssueは他のIssueの処理をブロックしない
- 各Issueの処理は独立したブランチで行う

### エラーハンドリング

- Issue取得に失敗した場合: エラーを報告して終了
- 個別Issueの処理に失敗した場合: スキップして次のIssueを処理
- コンフリクト検出に失敗した場合: 手動確認を推奨
- テスト失敗した場合: レポートに記録して続行

### レポートの保存先

- Phase 1レポート: `.dcs/batch-issues/{{timestamp}}_analysis.md`
- Phase 2レポート: `.dcs/batch-issues/{{timestamp}}_progress.md`
- Phase 3レポート: `.dcs/batch-issues/{{timestamp}}_integration.md`

### 音声通知のタイミング

- Phase開始時: 「フェーズN を開始するのだ」
- 各Issue処理完了時: 「イシューN番、完了なのだ」
- Phase完了時: 「フェーズN、完了なのだ」
- エラー発生時: 「エラーが発生したのだ」
- 全体完了時: 「バッチ処理、全て完了なのだ」

### ユーザ承認が必要なポイント

1. Phase 1完了時: 並列実行グループの承認
2. Phase 2開始前: 実行するIssueの最終確認
3. Phase 3のマージ順序: マージ実行前の承認
4. 手動対応項目: 対応方針の確認

### 禁止事項

- ユーザの承認なしにmainブランチへのマージを実行しない
- `git push --force` を使用しない
- `git rebase` を使用しない
- 機密情報を含むIssueを自動処理しない
- 処理中のIssueのステータスを勝手に変更しない（close等）
