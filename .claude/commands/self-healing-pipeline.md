# self-healing-pipeline

## 目的

PR作成前にコードベースの品質を自動検証し、問題があれば自動修正を試行するセルフヒーリングパイプライン。以下の3つのフェーズで構成される:

1. **Flakyテスト検出**: テストを実行し、不安定なテストを検出・記録
2. **サブエージェント実行リトライ**: 失敗したサブエージェントタスクの自動リトライ指針
3. **完了前検証スイープ**: PR作成前にテスト・型チェック・リント・ビルドを一括実行し、失敗項目を自動修正

## 前提条件

- プロジェクトルートに `package.json` が存在する
- `pnpm` がインストールされている
- 以下のコマンドが利用可能:
  - `pnpm test -- --run`（ユニットテスト）
  - `pnpm typecheck`（型チェック）
  - `pnpm lint`（リント）
  - `pnpm build`（ビルド）

## 実行手順

### Phase 1: Flakyテスト検出

Flakyテスト（実行ごとに成功/失敗が変わる不安定なテスト）を検出し、レポートを生成する。

#### Step 1.1: 初回テスト実行

1. VOICEVOX音声通知: 「フレイキーテスト検出を開始するのだ」（speaker=3, speedScale=1.3）
2. `pnpm test -- --run` を実行する（タイムアウト: 120000ms）
3. 実行結果を記録する:
   - 全テストがパスした場合 → Step 1.4へスキップ
   - 失敗テストがある場合 → 失敗テスト名のリストを記録し、Step 1.2へ

#### Step 1.2: 失敗テストのリトライ（最大3回）

失敗した各テストに対して以下を実行する:

1. リトライカウンターを初期化（retry_count = 0, max_retries = 3）
2. 失敗テストを個別に再実行する:
   - `pnpm test -- --run <テストファイルパス>`
3. 各リトライの結果を記録する:
   - **成功した場合**: 「Flaky候補」としてマーク
   - **失敗した場合**: retry_count をインクリメント
4. retry_count < max_retries の間、手順2-3を繰り返す

#### Step 1.3: Flakyテストの判定とレポート

リトライ結果に基づいてテストを分類する:

| 分類 | 条件 | 対応 |
|------|------|------|
| **Flaky** | 3回中1回以上成功、1回以上失敗 | レポートに記録、警告として扱う |
| **真の失敗** | 3回すべて失敗 | 修正が必要、Phase 3で対処 |
| **安定パス** | 3回すべて成功 | 初回の失敗は環境要因と判断 |

`docs/reports/flaky-tests.md` にレポートを出力する:

```markdown
# Flakyテストレポート

**検出日時**: YYYY-MM-DD HH:mm:ss
**実行環境**: Node.js vX.X.X / pnpm vX.X.X

## 検出されたFlakyテスト

| テストファイル | テスト名 | 成功回数/実行回数 | 備考 |
|---------------|---------|------------------|------|
| path/to/test | テスト名 | 2/3 | 初回のみ失敗 |

## 真の失敗テスト

| テストファイル | テスト名 | エラー概要 |
|---------------|---------|-----------|
| path/to/test | テスト名 | エラーメッセージ |

## 安定パステスト（初回のみ失敗）

| テストファイル | テスト名 | 推定原因 |
|---------------|---------|---------|
| path/to/test | テスト名 | 環境要因/タイミング |
```

#### Step 1.4: Phase 1完了

- VOICEVOX音声通知: 結果に応じたメッセージ
  - Flakyなし: 「テストは全て安定しているのだ」
  - Flakyあり: 「フレイキーテストを検出したのだ。レポートを確認してほしいのだ」
  - 真の失敗あり: 「失敗テストがあるのだ。修正が必要なのだ」

### Phase 2: サブエージェント実行リトライ指針

Agentツール（TaskCreate等）でサブエージェントを起動する際の自動リトライガイドライン。

#### リトライポリシー

| 項目 | 値 |
|------|-----|
| 最大リトライ回数 | 3回 |
| リトライ対象 | サブエージェントのタスク失敗、タイムアウト |
| リトライ非対象 | ユーザー入力エラー、権限不足、明確なロジックエラー |

#### リトライ手順

1. **1回目の実行**: 通常通りサブエージェントを実行
2. **失敗時の分析**: エラーメッセージを解析し、リトライ可能か判断
   - リトライ可能: ネットワークエラー、タイムアウト、一時的な環境問題
   - リトライ不可: コードの論理エラー、型エラー、権限不足
3. **2回目のリトライ**: エラーの原因を踏まえてプロンプトを調整し再実行
4. **3回目のリトライ**: さらにプロンプトを簡略化・分割して再実行
5. **3回失敗時のエスカレーション**:
   - VOICEVOX音声通知: 「サブエージェントが3回失敗したのだ。手動対応が必要なのだ」
   - 失敗の詳細をユーザーに報告
   - 手動修正の提案を提示

#### リトライ判定フロー

```
サブエージェント実行
  → 成功 → 完了
  → 失敗 → エラー分析
    → リトライ不可（論理エラー等） → エスカレーション
    → リトライ可能 → リトライ（最大3回）
      → 成功 → 完了
      → 3回失敗 → エスカレーション（音声通知 + 詳細報告）
```

### Phase 3: 完了前検証スイープ

PR作成前に全品質チェックを一括実行し、失敗した項目の自動修正を試行する。

#### Step 3.1: 検証スイープの実行

1. VOICEVOX音声通知: 「完了前検証スイープを開始するのだ」（speaker=3, speedScale=1.3）
2. 以下の4つのチェックを順番に実行する:

| 順序 | チェック | コマンド | タイムアウト |
|------|---------|---------|-------------|
| 1 | ユニットテスト | `pnpm test -- --run` | 120000ms |
| 2 | 型チェック | `pnpm typecheck` | 120000ms |
| 3 | リント | `pnpm lint` | 120000ms |
| 4 | ビルド | `pnpm build` | 120000ms |

3. 各チェックの結果を記録する:
   - PASS: 成功
   - FAIL: 失敗（エラー出力を保存）

#### Step 3.2: 失敗項目の自動修正試行

失敗した項目に対して、以下の順序で自動修正を試みる:

##### リントエラーの自動修正

1. `pnpm lint:fix` を実行
2. 再度 `pnpm lint` で確認
3. 修正できなかったエラーを記録

##### 型エラーの自動修正

1. エラーメッセージからエラー箇所（ファイル・行番号）を特定
2. エラー内容を分析し、修正可能か判断:
   - 型の不一致 → 適切な型アノテーションを追加
   - 未定義プロパティ → 型定義の確認・修正
   - インポートエラー → インポートパスの修正
3. 修正を適用し、`pnpm typecheck` で再確認

##### テストエラーの自動修正

1. エラーメッセージからテスト失敗箇所を特定
2. テストコードまたは実装コードのどちらに問題があるか分析
3. 明確なバグ（タイポ、パス間違い等）は自動修正
4. ロジックの変更が必要な場合はユーザーに報告

##### ビルドエラーの自動修正

1. エラーメッセージから原因を特定
2. インポートエラー、型エラー等は上記の手順で修正
3. 設定ファイルの問題は修正提案をユーザーに報告

#### Step 3.3: 修正後の再検証

1. 自動修正を行った場合、Step 3.1の全チェックを再実行する
2. 再検証は最大2回まで（無限ループ防止）
3. 再検証結果を記録する

#### Step 3.4: 最終判定

| 状態 | 判定 | アクション |
|------|------|-----------|
| 全チェックPASS | PR作成許可 | 「全チェックパスしたのだ。PR作成可能なのだ」 |
| 一部FAILだが軽微 | 条件付き許可 | ユーザーに状況を報告し判断を委ねる |
| 重大なFAIL残存 | PR作成不可 | 「未解決の問題があるのだ。修正が必要なのだ」 |

#### Step 3.5: 検証結果のサマリー

検証結果をユーザーに報告する:

```
## 完了前検証スイープ結果

| チェック | 結果 | 備考 |
|---------|------|------|
| ユニットテスト | PASS/FAIL | 詳細 |
| 型チェック | PASS/FAIL | 詳細 |
| リント | PASS/FAIL | 詳細 |
| ビルド | PASS/FAIL | 詳細 |

自動修正: X件実施（成功: Y件、失敗: Z件）
再検証: N回実施
最終判定: PR作成許可/不可
```

VOICEVOX音声通知: 最終結果に応じたメッセージ（speaker=3, speedScale=1.3）

## 全体フロー

```
パイプライン開始
  │
  ├─ Phase 1: Flakyテスト検出
  │   ├─ テスト実行
  │   ├─ 失敗テストのリトライ（最大3回）
  │   ├─ Flaky判定・レポート出力
  │   └─ 音声通知
  │
  ├─ Phase 2: サブエージェントリトライ指針（実行時に参照）
  │   ├─ リトライポリシーの適用
  │   ├─ 失敗時のエラー分析
  │   └─ エスカレーション通知
  │
  └─ Phase 3: 完了前検証スイープ
      ├─ 4つのチェック一括実行
      ├─ 失敗項目の自動修正試行
      ├─ 修正後の再検証（最大2回）
      ├─ 最終判定
      └─ 音声通知
```

## エラーハンドリング

### パイプライン自体のエラー

| エラー | 対応 |
|--------|------|
| pnpmコマンドが見つからない | 「pnpmがインストールされていないのだ」と通知して終了 |
| package.jsonが存在しない | 「プロジェクトルートにパッケージジェイソンがないのだ」と通知して終了 |
| タイムアウト | タイムアウトしたチェックを記録し、次のチェックに進む |
| ディスク容量不足 | 「ディスク容量が不足しているのだ」と通知して終了 |

### 自動修正のセーフガード

- 自動修正はPhase 3のみで実施（Phase 1では検出のみ）
- 1回の修正で変更するファイルは最大10ファイルまで
- 修正前に `git diff` で変更内容を確認可能な状態を維持
- 大規模な変更（50行以上の変更）が必要な場合はユーザーに確認

| タイミング | メッセージ例 |
|-----------|------------|
| パイプライン開始 | 「セルフヒーリングパイプラインを開始するのだ」 |
| Phase 1開始 | 「フレイキーテスト検出を開始するのだ」 |
| Phase 1完了 | 「テスト検出が完了したのだ」 |
| Phase 3開始 | 「完了前検証スイープを開始するのだ」 |
| 自動修正中 | 「自動修正を試みているのだ」 |
| 全チェックPASS | 「全チェックパスしたのだ。ピーアール作成可能なのだ」 |
| 失敗残存 | 「未解決の問題があるのだ。確認してほしいのだ」 |
| サブエージェント失敗 | 「サブエージェントが失敗したのだ。手動対応が必要なのだ」 |

## 注意事項

- Phase 1とPhase 3は独立して実行可能。必要に応じて個別にPhaseを指定して実行できる
- Phase 2はサブエージェント実行時に参照するガイドラインであり、単独実行はしない
- Flakyテストレポート（`docs/reports/flaky-tests.md`）は追記モードで更新し、過去の記録を保持する
- 自動修正は保守的に行い、不確実な修正はユーザーに確認を求める
- ビルドやテストのキャッシュが問題を起こしている可能性がある場合は、キャッシュクリア後に再実行を検討する
