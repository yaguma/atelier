# レビュアー出力フォーマット

## 概要

各レビュアーはTeam Leadへの報告時にこのフォーマットに従う。
SendMessageで送信する際、以下の構造化されたフォーマットで結果を報告する。

---

## 出力フォーマット

```markdown
# [観点名] Review Report

## Summary
- レビュー対象ファイル数: N
- 変更行数: +N / -N
- 検出件数: Critical N件 / Warning N件 / Info N件

## Critical Issues

### [C-連番] タイトル
- **カテゴリ**: C1〜C5（severity-definitions.md参照）
- **ファイル**: `path/to/file.ts:行番号`
- **問題**: 問題の説明
- **根拠**: 参照ルール名 or 技術的根拠
- **修正案**: 具体的な修正方法
- **影響範囲**: 影響を受ける他のコード

## Warning Issues

### [W-連番] タイトル
- **カテゴリ**: W1〜W5
- **ファイル**: `path/to/file.ts:行番号`
- **問題**: 問題の説明
- **根拠**: 参照ルール名 or 技術的根拠
- **修正案**: 具体的な修正方法

## Info Issues

### [I-連番] タイトル
- **カテゴリ**: I1〜I3
- **ファイル**: `path/to/file.ts:行番号`
- **提案**: 改善提案の内容

## Good Points

- 良い点を1〜3個挙げる（任意）

## Cross-Cutting Notes

- 他の観点との関連で気づいた点があれば記載（任意）
```

---

## 注意事項

### ファイル参照
- 問題のあるコードは必ず**ファイルパスと行番号**を明記する
- 行番号はPR差分ではなく、現在のファイルの行番号を使用する

### 根拠の記載
- プロジェクトのルールファイル（`.claude/rules/*.md`）を参照する場合はファイル名を明記
- 一般的なベストプラクティスの場合はその理由を簡潔に説明

### 差分に集中
- PR差分に含まれる変更のみをレビュー対象とする
- 既存コードの問題は「Cross-Cutting Notes」に記載し、別途Issue化を提案

### 重複回避
- 他のレビュアーと観点が重複する可能性がある場合は、自分の専門領域の視点で記述
- 統合はTeam Leadが行うため、重複を恐れず報告してよい
