# コンセプトデザイン・フレームワーク詳細

このドキュメントは、ゲームコンセプトを言語化するための5ステップフレームワークの詳細を記載する。

---

## Step 1: ワンセンテンス・コンセプト

### 目的

「世界観を考える」ことではない。**判断・選択・失敗を含んだ体験を言語化すること**。

「どんな遊びを、どんな密度で、どんな人に届けるか」を決める工程。

### 必須フォーマット

```
【誰が】【何をして】【どんな判断を繰り返し】【どんな感情になるゲームか】
```

### 良い例

- 「プレイヤーが限られたカードから行動を選び、短い判断の積み重ねで生き残るゲーム」
- 「探索者が未知のダンジョンで進退を判断し、欲張りすぎると全てを失う緊張感のあるゲーム」

### 悪い例（禁止）

- 「雰囲気重視のローグライク」→ 抽象語だけ
- 「戦略性の高いバトル」→ 判断・行動が入っていない
- 「美しい世界を探索する」→ 体験が曖昧

### ルール

- 抽象語だけの文は禁止
- 判断・行動が入っていない文はNG
- 1文で収まらなければ、まだ収束できていない証拠

---

## Step 2: ターゲット体験の制限

### なぜ必要か

すべて盛ると**設計が破綻**する。「これはやらない」を決めることで、実現可能な設計になる。

### 必須項目

#### プレイ時間

| 項目 | 記入欄 |
|------|--------|
| 1プレイ | ◯分 |
| セッション | ◯分 |

#### 複雑さレベル

- 初見で理解できるか？
- チュートリアル必須か？
- ルール説明に何分かかるか？

#### 操作密度

- 忙しい（リアルタイム・APM重視）
- 落ち着いて考える（ターン制・戦略重視）
- その中間

### やらないことリスト（例）

- 一本道ストーリーはやらない
- リアルタイム操作はしない
- PvP要素は入れない
- 課金要素は含めない
- 音声・フルボイスは入れない

## Step 3: 参考ゲームの分解

### NGなやり方

「Slay the Spire っぽい」「原神みたいな」

→ これではコピーになる。設計意図が見えない。

### OKなやり方

参考ゲームを**要素単位で分解**し、取り入れるものと捨てるものを明確にする。

### 分解テーブル

| 観点 | 取り入れる | 捨てる | 理由 |
|------|-----------|--------|------|
| 判断頻度 | | | |
| ランダム性 | | | |
| 永続成長 | | | |
| リソース管理 | | | |
| 失敗ペナルティ | | | |
| プレイヤースキル依存度 | | | |
| 情報公開度 | | | |

### 観点の補足

- **判断頻度**: 1分間に何回判断するか
- **ランダム性**: 低=将棋、中=ポーカー、高=スロット
- **永続成長**: ラン間で何が残るか
- **リソース管理**: HP/MP/ゴールド/時間など
- **失敗ペナルティ**: ロストの重さ
- **プレイヤースキル依存度**: 上達余地
- **情報公開度**: 見える情報 vs 隠された情報

## Step 4: 体験の強弱カーブ

### 目的

「退屈な谷」を作らない。プレイ開始から終了まで、判断の重さがどう変化するかを設計する。

### 書き方

```
開始：理解フェーズ（迷わない、安全な選択肢がある）
序盤：選択肢が増える（少しずつ判断が複雑になる）
中盤：判断が難しくなる（トレードオフが生まれる）
終盤：覚悟を迫られる選択（失敗のリスクが高まる）
クライマックス：全てを賭ける判断
```

### 注意点

- この時点では演出・ストーリー不要
- **判断の重さ**だけ考える
- 「何分後にどんな判断をさせるか」で考える

### 具体例

```
00:00-02:00 チュートリアル的な判断（失敗しても痛くない）
02:00-10:00 リソース獲得と消費のバランス判断
10:00-15:00 複数の選択肢からの取捨選択
15:00-20:00 蓄積したものを賭ける大きな判断
20:00-25:00 最終決戦（一発勝負）
```

## Step 5: コンセプト検証質問

### 目的

自分に嘘をつかない。「なんとなく面白そう」で終わらせない。

### 検証質問リスト

以下に**YES/NO**で答える：

1. **説明可能性**: 1プレイを30秒で説明できるか？
2. **判断中心**: 面白さを「操作」ではなく「判断」で説明できるか？
3. **リプレイ性**: 失敗しても次を試したくなる理由があるか？
4. **差別化**: 既存ゲームと何が違うか1文で言えるか？
5. **実現可能性**: 今の技術・リソースで作れるか？

### 判定

- **全てYES**: Step 2（技術設計）に進む
- **1つでもNO**: Step 1からやり直し

### NOの場合の対処

| 質問 | NOの原因 | 対処法 |
|------|---------|--------|
| 説明可能性 | コンセプトが複雑すぎる | 要素を削る |
| 判断中心 | アクション寄りになっている | 判断ポイントを増やす |
| リプレイ性 | 失敗が痛すぎる/学びがない | ペナルティ/報酬を調整 |
| 差別化 | 既存ゲームのコピー | Step 3をやり直す |
| 実現可能性 | スコープが大きすぎる | MVPを定義する |
