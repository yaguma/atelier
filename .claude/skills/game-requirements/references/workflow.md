## 1. プレイサイクル定義

### 目的
「1回のプレイで何を繰り返すか」を固定する。

### フォーマット
```
① 状況提示 - 何が見える？何が起きている？
② プレイヤーの選択 - 何を選ぶ？何を決める？
③ 結果反映 - 世界がどう変わる？
④ フィードバック - どう伝わる？（演出・音・数値）
→ ①に戻る
```

### ヒアリング質問
- 「プレイヤーは毎ターン何をするの？」
- 「1回の行動にかかる時間は？」
- 「行動の結果はすぐ分かる？遅延する？」

### チェックポイント
- ループが**30秒〜1分**で回るか
- 各フェーズが明確に分離しているか
- フィードバックが十分に気持ちいいか

## 2. 勝敗条件の明文化

### 目的
ゲームの終了条件を明確にする。

### 質問リスト
- 「プレイヤーは何を達成すれば勝ち？」
- 「失敗はいつ確定する？」
- 「途中撤退はある？」
- 「引き分けはある？」

### 書き方
```markdown
- 勝利条件: 最終ステージをクリア
- 敗北条件: HPが0になる
- 継続不能条件: デッキが尽きる
- 撤退条件: いつでも可能（獲得報酬は半減）
```

### 注意点
- 曖昧だと設計が破綻する
- 複数の勝利条件がある場合は優先順位を明記
- 「詰み」状態の定義も重要

## 3. プレイヤー操作の洗い出し

### 目的
プレイヤーができることを**動詞**で列挙する。

### ヒアリング質問
- 「プレイヤーは何ができる？」
- 「戦闘中と非戦闘中で違う？」
- 「やらなくてもいい操作は？」

### 列挙例
```
- 選ぶ（カード、行き先、装備）
- 使う（スキル、アイテム）
- 捨てる（カード、アイテム）
- 強化する（装備、スキル）
- 回避する（攻撃、罠）
- 配置する（ユニット、建物）
- 移動する（キャラクター）
```

### 注意点
- UIや操作方法はまだ考えない
- 行為そのものに集中
- 「しない」選択肢も操作に含める

## 4. ゲーム要素定義

### 目的
システム化できる「モノ」を洗い出す。

### 代表的なエンティティ
- プレイヤー
- 敵
- カード / スキル / アビリティ
- ステータス（バフ/デバフ）
- リソース（HP / マナ / 時間 / お金）
- マップ / ステージ
- アイテム / 装備

### 各要素に対して定義すること
```markdown
### {要素名}
- 持っているもの（状態）: HP, 攻撃力, 所持カード
- 変化するもの: HP減少, レベルアップ, バフ付与
- 関係性: プレイヤーが所有, 敵に影響を与える
```

### ヒアリング質問
- 「ゲームに登場するモノは？」
- 「そのモノは何を持っている？」
- 「何が変化する？」

## 5. 変数・数値の仮置き

### 目的
「調整できる形」に落とす。

### フォーマット
```markdown
| 変数名 | 範囲 | 初期値 | 説明 |
|-------|------|-------|------|
| プレイヤーHP | 10〜100 | 50 | プレイヤーの体力 |
| 手札枚数 | 3〜7 | 5 | 毎ターンの手札 |
| ターン数上限 | 5〜20 | 10 | 1ステージのターン数 |
```

### ヒアリング質問
- 「大体どのくらいの数値をイメージしている？」
- 「最小・最大はどのくらい？」
- 「調整したくなりそうな数値は？」

### 注意点
- 正解じゃなくていい
- **数値が存在すること**が重要
- 後で調整できるように範囲を持たせる

## 6. 想定プレイフロー

### 目的
実際に遊べるかを文章でシミュレーションする。

### フォーマット
```markdown
### 1ターン目
- 状況: 手札3枚、敵HP10
- 行動: 攻撃カードを使用
- 結果: 敵HPが5減少、手札が2枚に

### 2ターン目
- 状況: 手札4枚（1枚ドロー）、敵HP5
- 行動: 強攻撃カードを使用
- 結果: 敵を撃破、報酬獲得
```

### チェックポイント
- 破綻していないか（詰み状態が発生しないか）
- テンポは良いか
- 意思決定の余地があるか
- 想定外の行動パターンは？

### ヒアリング質問
- 「最初のターンは何が起きる？」
- 「典型的な勝ちパターンは？」
- 「典型的な負けパターンは？」
